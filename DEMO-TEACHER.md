# Демонстрация преподавателю: отказоустойчивость Swarm и синхронизация данных

Цель: показать, что при падении ноды с БД веб-сервис остаётся доступным, обращение к базе временно невозможно, затем за несколько секунд (до минуты) все восстанавливается, **все записи в базе на месте** (данные на NFS).

---

## Что нужно заранее

- Кластер из 3 нод (master, worker1, worker2), Swarm инициализирован.
- NFS настроен: `/mnt/mysqldata` на всех нодах (см. SETUP-FULL.md и NFS-SETUP.md).
- На всех нодах метка: `docker node update --label-add mysqldata=true NODE`.
- Стек задеплоен: `docker stack deploy -c docker-compose.yaml bank-guarantee`.
- Приложение открывается в браузере по `http://<IP-master>:3000`.

---

## Пошаговая демонстрация

### 1. Внести изменения в базу

Откройте веб-интерфейс приложения. Создайте несколько записей (клиенты, заявки и т.п.). Убедитесь, что данные отображаются - это "изменения в базу".

### 2. Узнать ноду, где крутится БД

На master выполните:

```bash
docker service ps bank-guarantee_db --no-trunc
```

Запомните значение в колонке **NODE** (например, `worker1`). Эту виртуалку нужно будет выключить.

### 3. Показать, что веб и БД работают

Обновите страницу приложения, откройте разделы с данными — всё загружается. Можно в терминале на master запустить:

```bash
while true; do curl -s -o /dev/null -w "%{http_code} " http://localhost:3000/api/bootstrap; sleep 2; done
```

Должны идти ответы **200**.

### 4. «Прибить» ноду с СУБД

Выключите виртуалку, на которой в шаге 2 была задача `bank-guarantee_db`:

- в VirtualBox: Машина -> Завершить работу -> Выключить,  
  или на самой ноде: `sudo poweroff`.

Через несколько секунд:

- В цикле `curl` появятся ошибки (500 или таймаут) — **веб доступен, но обратиться к базе нельзя**.
- На master: `docker node ls` — одна нода станет **Down** / **Unreachable**.
- `docker service ps bank-guarantee_db` — задача на упавшей ноде в состоянии Shutdown, затем появится **новая задача на другой ноде** (Running).

### 5. Подождать восстановления (несколько секунд — до минуты)

Swarm поднимет контейнер MySQL на другой ноде; данные те же, т.к. каталог `/mnt/mysqldata` общий (NFS).

- Снова выполните: `docker service ps bank-guarantee_db` — контейнер в состоянии **Running** на другой ноде.
- Обновите страницу приложения — данные снова отображаются, **все ранее сделанные записи на месте**.
- Цикл `curl` снова начнёт возвращать **200**.

### 6. Формулировка для преподавателя

> Мы внесли изменения в базу через веб-интерфейс. Затем отключили виртуалку с контейнером СУБД. Веб-сервис оставался доступным, но к базе обратиться было нельзя. Через несколько секунд Swarm перенес контейнер БД на другую ноду; данные на общем NFS - все записи восстановились, синхронизация и отказоустойчивость обеспечены.

## Если что-то пошло не так

- **Нода не помечена** - на master:  
  `docker node update --label-add mysqldata=true worker1` (и для worker2, master).
- **Нет сети** — `docker network create --driver overlay --attachable bank_guarantee_net`.
- **Нет образа backend** — на master в каталоге проекта:  
  `docker build -t bank-guarantee-backend:latest -f server/Dockerfile .`
- **БД не поднимается на другой ноде** - проверьте, что на worker’ах смонтирован NFS:  
  `df -h /mnt/mysqldata` и что каталог доступен для записи (права 999:999 на master в экспорте).

Подробная настройка: **SETUP-FULL.md**.

## Как сократить время восстановления (опционально)

- **Образы на всех нодах** — чтобы при переносе БД не тратить время на скачивание, один раз на **каждой** ноде выполните:
  ```bash
  sudo docker pull mysql:8.0
  sudo docker pull bank-guarantee-backend:latest
  ```
  (образ backend должен быть собран и сохранён/загружен на worker’ы, если БД может подняться там.)

- В **docker-compose** уже заданы: `restart_policy.delay: 1s` у БД, быстрый **healthcheck** и параметры MySQL для ускоренного старта.

- **Ускорить обнаружение падения ноды (в ~2 раза):** на master один раз выполните:
  ```bash
  sudo docker swarm update --dispatcher-heartbeat 2s
  ```
  По умолчанию Swarm реже проверяет ноды; при 2s падение обнаруживается за ~6–10 с вместо 15–30 с.
